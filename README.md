# Digital Heroes 🏌️‍♂️💚

> Play your game. Power a cause. Win rewards.

A subscription-driven web platform that combines **golf performance tracking**, **charity fundraising**, and a **monthly draw-based reward engine**. Members log their last 5 Stableford scores, support a charity of their choice, and enter a monthly draw where every rupee contributes to meaningful causes.

🔗 **Live demo:** https://golf-draw.lovable.app

---

## ✨ Features

### Public site
- **Landing page** — hero, "How it works", featured charities, pricing preview
- **Charity directory** — searchable, filterable, with detail pages
- **Pricing page** — Monthly (₹499) & Yearly (₹4,999) plans
- **How It Works** — full explainer page

### Authentication
- Email + password sign-up / sign-in (Lovable Cloud Auth)
- Auto-confirm enabled — no email verification step
- Role-based access via a separate `user_roles` table (`admin` / `subscriber`)
- Protected routes for Dashboard, Checkout, and Admin

### User Dashboard
- **Score entry** — Stableford scores 1–45, date-restricted
- **5-score rolling logic** — DB trigger auto-deletes the oldest score when a 6th is added
- **Charity selection** + adjustable contribution slider (10–100%)
- **Subscription status** with renewal date
- **Winning proof upload** for verification

### Admin Panel
- **Users tab** — promote/demote admins, view subscriptions
- **Charities tab** — create / edit / feature charities
- **Draw engine** — run **Random** or **Algorithmic** (frequency-weighted) draws
  - Simulate first → Publish when ready
  - Auto-computes Match-5 / Match-4 / Match-3 winners
  - Pool split: 40% Match-5, 35% Match-4, 25% Match-3
  - Jackpot rollover when no Match-5
- **Winners tab** — verify proof, mark payouts (pending / paid)

### Payments
- **Mock checkout flow** — no real payment gateway integration
- Creates an active subscription record on confirm
- All amounts in **Indian Rupees (₹)** with Indian numbering format

## 🗄️ Database Schema

| Table | Purpose |
|---|---|
| `profiles` | User profile, chosen charity & contribution % |
| `user_roles` | Separate role storage (`admin`, `subscriber`) — prevents privilege escalation |
| `subscriptions` | Plan, amount, status, renewal date |
| `charities` | Charity directory (name, category, image, featured flag) |
| `scores` | Stableford scores (1–45) with auto-trim trigger to last 5 |
| `draws` | Monthly draws — winning numbers, pool, rollover, simulation flag |
| `winners` | Match results, prize, verification & payout status |

Key DB objects:
- `has_role(user_id, role)` — `SECURITY DEFINER` function used in all RLS policies
- `enforce_five_scores()` — trigger keeping only the latest 5 scores per user
- `handle_new_user()` — creates profile + default `subscriber` role on signup

## 🚀 Getting Started (Local)

### Prerequisites
- Node.js 18+ and `bun` (or npm)

### Setup
```bash
# Clone
git clone <your-repo-url>
cd <repo>

# Install
bun install

# Run dev server
bun run dev
```

Open http://localhost:5173

### Environment
The `.env` file is auto-generated by Lovable Cloud and contains:
```
VITE_SUPABASE_URL=...
VITE_SUPABASE_PUBLISHABLE_KEY=...
VITE_SUPABASE_PROJECT_ID=...
```
> Don't edit `.env`, `src/integrations/supabase/client.ts`, or `src/integrations/supabase/types.ts` — they're managed automatically.

## 👤 First-time Usage

1. **Sign up** at `/auth?mode=signup`
2. Choose a **plan** at `/pricing` → mock-pay at `/checkout`
3. From `/dashboard`:
   - Pick your **charity** + contribution %
   - Enter up to **5 Stableford scores**
4. Visit `/admin` to manage charities, run draws, and verify winners.

## 🎲 Draw Engine Logic

```ts
// Random — 5 distinct numbers between 1 and 45
randomDraw()

// Algorithmic — weighted toward most-frequent user scores (+1 base so all numbers possible)
algorithmicDraw(allScores)

// Match users' last-5 score sets against winning numbers
computeMatches(winning, entries) // → { match_5, match_4, match_3 }
```

Pool split: **Match-5: 40%**, **Match-4: 35%**, **Match-3: 25%**.
If no Match-5 winners, that share **rolls over** to next month.

## 🔐 Security Notes

- All tables use **Row Level Security**
- Admin checks use the `has_role()` SECURITY DEFINER function (no recursive RLS)
- Roles are **never** stored on the profiles table
- The published anon key is safe to ship in client code
